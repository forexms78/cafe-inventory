# 코드 한 줄 안 쓰고 실제 카페에 투입한 재고관리 앱을 하루 만에 만든 방법

> AI 에이전트에게 "어떻게"를 전부 맡겼더니 생긴 일

---

나는 카페를 운영하는 지인의 문제를 해결하고 싶었다.

매일 아침 매니저들이 세 곳(매장 카운터 / 팬트리 / 사무실)을 직접 돌며 수기로 재고를 체크한다. 어떤 품목이 부족한지 한눈에 볼 방법이 없고, 부족한 걸 발견하면 쿠팡을 따로 열어 검색해야 한다. 불편한 걸 알면서도 "그냥 해왔던 방식"이라는 이유로 계속 반복 중이었다.

웹앱을 만들면 해결된다. 근데 나는 이걸 **직접 코딩하지 않고** 만들기로 했다.

---

## AI 에이전트와 일하는 방식

내가 사용한 건 Claude Code다. 터미널에서 돌아가는 AI 에이전트인데, 파일을 읽고 쓰고, 명령어를 실행하고, git까지 다룬다.

내 역할은 딱 하나였다. **"무엇을 만들지"만 결정하는 것.**

"어떻게"는 전부 AI에게 맡겼다.

예를 들면 이런 식으로 대화했다.

```
나: "모바일에서 화면이 잘려. 상세 제품명이 길어서 재고 수정이 어려워"

AI: ItemRow.tsx 수정 — 모바일에서 product_name hidden sm:inline 처리,
    테이블에 overflow-x-auto 래퍼 추가, min-w-[500px] 고정
```

```
나: "유통기한 클릭하면 캘린더 뜨게 해줘"

AI: ExpiryModal.tsx 신규 생성, input[type=date] 적용,
    기존 YYMMDD 형식 → YYYY-MM-DD 자동 변환 로직 포함
```

기술 용어를 쓰지 않았다. 문제 상황을 설명했을 뿐이다.

## 뭘 만들었나

**카페 재고 관리 웹앱.** 실제 운영 중인 디저트 카페에 투입됐다.

핵심 플로우는 하나다.

```
재고 입력 → 부족 자동 감지 → 클릭 1회 → 쿠팡/네이버쇼핑 즉시 검색
```

구현된 기능들:

- **3위치 통합 재고** — 매장/팬트리/사무실 각각 입력, 위험/주의/정상 자동 분류
- **부족 알림 대시보드** — 전 카테고리 부족 현황을 한 화면에
- **품목 검색** — 타이핑하면 카테고리 자동 전환 + 해당 행으로 스크롤
- **드래그앤드롭 정렬** — 자주 쓰는 품목 위로
- **유통기한 캘린더** — 날짜 클릭으로 입력
- **낙관적 업데이트 + 롤백** — UI 즉시 반영, API 실패 시 자동 복원
- **토스트 메시지** — 저장 여부 즉시 피드백

기술 스택은 AI가 골랐다. Next.js 16, Supabase, Tailwind, shadcn/ui, @dnd-kit, sonner. 별도 백엔드 서버 없이 Next.js Route Handlers로 API를 처리했다. 합리적인 선택이었다.

## 실제로 겪은 빌드 오류

AI가 완벽하지 않다는 걸 알고 시작했다. 예상대로 오류가 났다.

**오류 1 — 누락 파일**

Vercel 빌드 중에 터졌다.

```
Module not found: Can't resolve '@/components/PriceCompareModal'
```

원인은 간단했다. AI가 로컬에서 파일을 만들었는데 git에 안 올라간 것. `git status`를 보니 untracked 상태였다. 커밋하니 해결됐다.

**오류 2 — Props 타입 불일치**

```
Property 'onProductNameChange' is missing in type
```

로컬에 커밋 안 된 변경사항이 있는 상태에서 일부 파일만 먼저 올라가면서 타입이 안 맞게 된 케이스. 미커밋 파일 전부 반영하니 해결됐다.

두 케이스 모두 에러 로그를 AI에게 그대로 붙여넣으니 원인을 찾고 수정까지 했다.

## 낙관적 업데이트 — 속도 vs 정합성 딜레마

이 프로젝트에서 가장 흥미로운 기술적 결정이었다.

재고 +1 버튼을 누를 때 두 가지 방법이 있다.

**방법 A (API 대기):** 버튼 클릭 → API 호출 → 응답 받으면 UI 업데이트
→ 항상 정확하지만 느리다. 네트워크 지연이 고스란히 체감된다.

**방법 B (낙관적 업데이트):** 버튼 클릭 → UI 즉시 업데이트 → 백그라운드에서 API 호출
→ 빠르지만, API가 실패해도 UI는 바뀐 상태로 남는다.

AI가 선택한 방법은 B + 롤백이다.

```typescript
const handleStockChange = async (id, field, value) => {
  const prev = items.find(i => i.id === id)?.[field] ?? 0;

  // 1. UI 먼저 변경
  setItems(current => current.map(i => i.id === id ? { ...i, [field]: value } : i));

  const res = await fetch('/api/items', { method: 'PATCH', ... });

  if (!res.ok) {
    // 2. 실패 시 원래 값으로 롤백
    setItems(current => current.map(i => i.id === id ? { ...i, [field]: prev } : i));
    toast.error('저장 실패. 다시 시도해주세요.');
  } else {
    toast.success('저장됨');
  }
};
```

속도 저하 없이 데이터 정합성도 잡는 구조다. 카페 와이파이 환경에서는 실패 케이스가 드물지만, 실패했을 때 데이터가 꼬이면 안 되는 시스템이라 넣었다.

## AI와 개발하면서 배운 것

**What만 말하고 How는 맡겨라.**

"반응형 레이아웃 만들어줘"가 아니라 "모바일에서 화면이 잘려서 재고 수정이 어렵다"고 말해라. 문제를 설명하면 AI가 적합한 해결책을 찾는다. 기술적인 지시는 오히려 AI의 판단을 제한한다.

**에러 로그를 통째로 붙여넣어라.**

"왜 이런 에러가 나는지"를 분석하려 하지 말고 그냥 붙여넣어라. AI가 더 빠르게 원인을 찾는다.

**작게 쪼개서 반복해라.**

"전체 모바일 대응 다 해줘"보다 하나씩 요청하면 결과가 훨씬 낫다. 큰 덩어리로 요청할수록 놓치는 부분이 생긴다.

**git 상태를 주시해라.**

AI는 파일을 편집하지만 커밋은 명시적으로 요청해야 한다. `git status`를 자주 확인하거나, 작업 후 "커밋하고 푸쉬해줘"를 습관적으로 붙여라.
