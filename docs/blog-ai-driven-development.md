# AI 에이전트와 단 하루 만에 실전 투입 재고관리 앱을 만든 방법

> "코드 한 줄 직접 안 쓰고 실제 운영 중인 카페에 투입했습니다"

---

## 시작은 현실적인 불편함이었다

실제로 운영 중인 디저트 카페에는 재고가 세 곳에 흩어져 있다.

- 매장 카운터
- 팬트리
- 사무실

매일 아침 매니저들이 세 곳을 직접 돌며 수기로 재고를 확인하고, 부족한 품목을 발견하면 쿠팡이나 네이버쇼핑을 따로 열어 주문한다. 번거롭고 느리고, 타이밍을 놓치는 일도 잦다.

나는 이 문제를 해결할 웹앱을 만들기로 했다.
단, **코드를 직접 쓰지 않고**, AI 에이전트(Claude Code)에게 맡기는 방식으로.

---

## 내가 한 것 vs AI가 한 것

이 프로젝트에서 내가 한 일은 딱 두 가지다.

**내가 한 것:**
- "무엇을 만들지" 결정 (요구사항 정의)
- 결과물 검토 및 피드백

**AI 에이전트가 한 것:**
- 기술 스택 결정
- 컴포넌트 설계 및 구현
- API 설계 및 구현
- 버그 수정
- 배포 오류 디버깅

프롬프트는 이런 식이었다.

```
"재고관리 테이블에서 모바일 화면이 잘려.
상세 제품명은 모바일에서 숨기고, 가로 스크롤 추가해줘"
```

```
"유통기한을 클릭하면 캘린더 뜨게 해줘"
```

```
"재고 변경하면 토스트 메시지 보여주고,
API 실패하면 원래 값으로 롤백해줘"
```

기술적인 지시는 없다. What만 말한다. How는 AI가 결정한다.

## 기술 스택 — AI가 선택한 구조

AI 에이전트가 선택한 스택은 다음과 같다.

| 영역 | 선택 | 이유 |
|------|------|------|
| Frontend | Next.js 16 App Router | SSR + API Routes 통합으로 별도 백엔드 불필요 |
| Styling | Tailwind CSS + shadcn/ui | 빠른 프로토타이핑, 접근성 컴포넌트 |
| Database | Supabase (PostgreSQL) | 인증 + DB + API를 한 번에 |
| 드래그앤드롭 | @dnd-kit | 모바일 터치 지원 |
| 토스트 | sonner | 경량, 스타일 커스텀 용이 |
| 배포 | Vercel | git push → 자동 배포 |

별도 백엔드 서버 없이 Next.js Route Handlers로 API를 처리하는 구조를 선택했다. 소규모 앱에 최적화된 판단이다.

## 개발 흐름 — 대화로 기능을 쌓아가다

### 1단계 — 핵심 기능 구현

첫 번째 프롬프트로 기본 뼈대가 완성됐다.

- 7개 카테고리 탭
- 3위치 재고 입력 (+/- 버튼, 직접 입력)
- 최소 수량 기준 위험/주의/정상 자동 분류 (빨강/노랑/초록)
- Supabase CRUD API

### 2단계 — 사용성 개선

운영 중에 발견된 불편함을 하나씩 요청했다.

- "부족 품목 클릭하면 해당 행으로 이동해줘" → 카테고리 전환 + 스크롤 + 하이라이트
- "쿠팡 바로 연결해줘" → 제품명 등록 후 🛒 클릭으로 3개 쇼핑몰 즉시 검색
- "순서 바꾸고 싶어" → DnD 정렬 (@dnd-kit)

### 3단계 — 모바일 대응

실제 현장에서 스마트폰으로 사용하면서 문제가 생겼다.

```
"모바일에서 화면이 잘려. 상세 제품명이 너무 길어서
재고 수정이 어려워"
```

AI가 제안한 해결책:
- 상세명(product_name)은 모바일에서 `hidden sm:inline`으로 숨김
- 테이블 전체에 `overflow-x-auto` 가로 스크롤 래퍼 추가
- `min-w-[500px]`로 최소 너비 고정
- 재고 부족 알림: `grid-cols-1 sm:grid-cols-3` 반응형 그리드

### 4단계 — 데이터 안정성

```
"재고 변경할 때 저장됐는지 모르겠어.
그리고 실패하면 어떻게 돼?"
```

낙관적 업데이트(Optimistic Update) 패턴을 적용했다.

```typescript
const handleStockChange = async (id, field, value) => {
  const prev = items.find(i => i.id === id)?.[field] ?? 0;
  // 1. UI 먼저 변경 (빠른 반응성)
  setItems(current => current.map(i => i.id === id ? { ...i, [field]: value } : i));

  const res = await fetch('/api/items', { method: 'PATCH', ... });

  if (!res.ok) {
    // 2. 실패 시 원래 값으로 롤백
    setItems(current => current.map(i => i.id === id ? { ...i, [field]: prev } : i));
    toast.error('저장 실패. 다시 시도해주세요.');
  } else {
    toast.success('저장됨');
  }
};
```

속도 저하 없이 데이터 정합성을 보장하는 구조다.

## 실제로 겪은 빌드 오류들

AI가 완벽하진 않다. 중간에 이런 일도 있었다.

**문제 1 — 미커밋 파일**

```
Module not found: Can't resolve '@/components/PriceCompareModal'
```

AI가 로컬에서 작업한 파일을 git에 올리지 않은 채 배포한 경우였다. 원인을 파악하고 누락 파일을 커밋해서 해결했다.

**문제 2 — Props 타입 불일치**

```
Property 'onProductNameChange' is missing in type
```

로컬에 커밋되지 않은 변경사항이 있는 상태에서 일부 파일만 먼저 커밋되면서 타입이 맞지 않는 상황이 생겼다. 미커밋 파일 일괄 반영으로 해결했다.

이런 오류들이 나왔을 때 AI에게 에러 로그를 그대로 붙여넣으면 됐다. 원인 분석부터 수정까지 알아서 처리했다.

## AI 에이전트와 개발하는 방법 — 내가 배운 것

### 1. What만 말하고 How는 맡겨라

"모바일 대응해줘"가 아니라 "모바일에서 화면이 잘려서 재고 수정이 어렵다"고 말해라. 문제를 설명하면 AI가 적절한 해결책을 찾는다.

### 2. 에러 로그를 통째로 붙여넣어라

원인 분석을 AI에게 맡겨라. "왜 이런 에러가 나는지"보다 "에러 로그 여기 있어, 고쳐줘"가 훨씬 빠르다.

### 3. 한 번에 큰 덩어리보다 작게 반복해서

"전체 모바일 대응 다 해줘"보다 "1. 이름 잘림 해결, 2. 알림 레이아웃 수정, 3. 가로 스크롤 추가" 식으로 쪼개서 요청하면 결과 품질이 높다.

### 4. 커밋 상태를 주시해라

AI는 파일을 편집하지만 커밋은 하지 않는다 (명시적으로 요청하지 않으면). git status를 자주 확인하거나 AI에게 커밋까지 요청하는 습관을 들여라.

## 마무리

이 프로젝트는 "AI로 코딩하면 어디까지 가능한가"에 대한 나의 실험이었다.

결론은 명확하다. **실전 투입 가능한 수준의 웹앱을 코드 한 줄 안 쓰고 하루 만에 만들 수 있다.**

중요한 건 AI가 코드를 잘 짜는지가 아니다. 내가 문제를 얼마나 명확하게 정의하는지다.

AI는 How를 실행하는 엔진이다. What을 결정하는 건 여전히 사람의 몫이다.
